## Spacing Scale

Inherited from Phase 2. No changes.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding, stat card internal gaps |
| sm | 8px | Compact element spacing, dashboard grid gaps, notification item padding |
| md | 16px | Default element spacing, card padding, dashboard section padding |
| lg | 24px | Section padding, dashboard column gaps, dialog body padding |
| xl | 32px | Layout column gaps, page section margins |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level top/bottom spacing |

**Phase 8 exceptions:**

| Element | Value | Reason |
|---------|-------|--------|
| Dashboard stat card min-height | 96px | Stat cards need room for number + label without crowding |
| Notification bell dropdown width | 360px | Notification items need space for title + timestamp + action |
| Notification bell dropdown max-height | 480px | Cap dropdown height; scroll internally beyond 480px |
| Notification item touch target | 44px min-height | WCAG 2.5.5 Target Size for notification items |
| Dashboard section card grid gap | 16px | Tighter than xl to keep dashboard information-dense |
| "What changed" indicator dot size | 8px | Unread/changed indicator — small enough to not distract |
| Health indicator pill height | 24px | Section health pill badge (Good / Warning / Critical) |

## Color

Inherited from Phases 2–7. CSS custom properties from `app/globals.css` (oklch-based neutral palette).

| Role | Light Value | Dark Value | Usage |
|------|-------------|------------|-------|
| Dominant (60%) | `oklch(1 0 0)` (white) | `oklch(0.145 0 0)` (near-black) | Page background, dashboard content area |
| Secondary (30%) | `oklch(0.97 0 0)` (neutral-50) | `oklch(0.269 0 0)` (neutral-800) | Dashboard widget card backgrounds, notification dropdown background, section health card backgrounds |
| Accent (10%) | `oklch(0.205 0 0)` (near-black) | `oklch(0.922 0 0)` (near-white) | See reserved list below |
| Destructive | `oklch(0.577 0.245 27.325)` (red) | `oklch(0.704 0.191 22.216)` (red) | No destructive actions in Phase 8 |

All color values referenced via CSS custom properties only. Never hardcode oklch values directly.

**Accent reserved for these specific elements only in Phase 8:**
1. "View all" primary navigation CTA buttons within dashboard widgets (filled)
2. Active notification bell indicator dot (unread count badge on bell icon)
3. "What changed" indicator dot on section rows (unread/changed marker)
4. Active tab indicator in dashboard sub-tabs (e.g., "All / Unread" in notification dropdown)
5. Primary dashboard CTA button per role (one per dashboard — see copywriting contract)

**Do NOT use accent for:** health indicator pills (use semantic green/amber/red — see semantic colors below), stat card numbers (use foreground), notification body text, activity feed timestamps, badge pill text.

**Semantic colors — Phase 8 additions:**

| Semantic | CSS Variable / Value | Usage |
|----------|---------------------|-------|
| Health: Good | `oklch(0.527 0.154 150)` (green) bg / foreground | Section health "Good" pill — feedback below threshold, no open CRs |
| Health: Warning | `oklch(0.769 0.188 70)` (amber) bg / foreground | Section health "Warning" pill — feedback accumulating, open CRs |
| Health: Critical | `oklch(0.577 0.245 27)` (red) bg / foreground | Section health "Critical" pill — high-priority unresolved feedback, blocked CRs |
| Notification: Unread | `oklch(0.205 0 0)` (accent, near-black) | Unread notification dot, unread count badge |
| Notification: Read | `var(--muted-foreground)` | Read notification row text |
| What Changed dot | `oklch(0.527 0.154 150)` (green) | "What changed since last visit" dot on stakeholder dashboard |

**Existing semantic colors — all inherited, no redeclarations:**
- All FB status badge colors (Phase 4)
- All CR status badge colors (Phase 5)
- Border, muted foreground, ring (Phase 2)

**Health indicator implementation:** Use inline `<span>` with `bg-[color]/20 text-[color]` pattern (semitransparent background + solid text color), consistent with existing Phase 4/5 status badge treatment. Do NOT introduce new CSS variables for health colors — use Tailwind utility classes with the okclh values declared above as arbitrary values if not already in the token set.

## Component Inventory

### Global: Notification Bell

The notification bell is a persistent UI element present in the workspace header across all pages when authenticated. It is introduced in Phase 8.

**Focal point declaration:** Bell icon is the anchor. Unread count badge (accent dot or number badge) on the bell signals pending notifications. Click opens a dropdown. Eye lands on the unread items first (bold text), then reads down the list, then lands on the "Mark all as read" / "View all" footer row.

| Element | Component | Variant | Notes |
|---------|-----------|---------|-------|
| Bell trigger | `Button` | ghost, size="icon" | Bell icon (16px). `aria-label="Notifications"`. When unread count > 0: shows `Badge` with count as absolute overlay (top-right of button, 16px, accent background, white text, rounded-full) |
| Notification dropdown | Custom `Popover` (base-ui) | -- | Width: 360px. max-height: 480px. Internal scroll via `ScrollArea`. Triggered by bell button. Closes on outside click. |
| Dropdown header | Custom flex row | -- | "Notifications" (Heading 20px/600, left) + "Mark all as read" (Button, ghost, size="sm", right — only shown when unread > 0) |
| Notification tabs | `Tabs` | -- | 2 tabs: "All" / "Unread". Active tab: accent underline indicator. |
| Notification item | Custom list item | -- | 44px min-height. Two states: unread (bg-muted, title at 14px/600) and read (bg-transparent, title at 14px/400). Structure: [icon] [text col] [timestamp]. Clickable — routes to the relevant object. |
| Notification icon | lucide icon | -- | 16px. Per event type: MessageSquare (feedback status change), GitPullRequest (CR status change), BookOpen (version published), UserPlus (section assignment). Color: accent for unread, muted-foreground for read. |
| Notification title | `<p>` | 14px/600 (unread) or 14px/400 (read) | Truncated to 1 line |
| Notification body | `<p>` | 12px/400, muted-foreground | Truncated to 2 lines. Full text in tooltip on hover. |
| Notification timestamp | `<time>` | Label (12px/400), muted-foreground | Relative time (e.g., "2 minutes ago", "Yesterday") |
| Dropdown footer | Custom flex row | -- | "View all notifications" link → `/notifications` page. Styled as Button ghost, full-width, centered. Only shown when total count > 5. |
| Empty state | Custom div | -- | Bell icon (32px, muted-foreground), centered. "No notifications yet." (14px/400, muted-foreground) |
| Loading state | `Skeleton` | -- | 3 skeleton notification items (h-10 w-full with inner skeleton rows) |

**Notification item layout:**
```
Row (44px min-height, px-md py-sm):
  [Icon (16px, colored)] [
    Title (truncate 1 line, 14px/600 or /400)
    Body snippet (truncate 2 lines, 12px/400, muted-foreground)
  ] [Timestamp (12px/400, muted-foreground, ml-auto, shrink-0)]
```

## Accessibility Requirements

| Requirement | Implementation |
|-------------|----------------|
| Keyboard navigation | Bell trigger reachable via Tab. Dropdown items navigable with Arrow Up/Down. Dashboard widget rows navigable via Tab. |
| Focus management | After opening bell dropdown: first notification item receives focus. On dropdown close: focus returns to bell trigger. After "Mark all as read": focus stays on bell trigger. |
| Screen reader | Bell button: `aria-label="Notifications, {N} unread"` when unread > 0; `aria-label="Notifications"` when 0. Unread dot: `aria-label="Unread"`. Notification item: `role="button"` + `aria-label="{event title} — {relative time}"`. Stat cards: `aria-label="{stat value} {stat label}"`. Health pills: `aria-label="Section {name} health: {Good/Warning/Critical}"`. |
| Live region | Notification dropdown: `aria-live="polite"` on the count badge so screen readers announce count changes when dropdown is open. |
| Color contrast | All text meets WCAG 2.1 AA (4.5:1 body, 3:1 large text). Health semantic colors (green/amber/red) are used in combination with text labels, not color-only. |
| Motion | No animated transitions in dashboard beyond shadcn defaults. Skeleton shimmer respects `prefers-reduced-motion`. |
| "What changed" dot | Green dot alone is NOT the only indicator — always accompanied by text copy. |

### Dashboard Routing Note

The `/dashboard` route was established in Phase 2 as a navigation anchor. Phase 8 implements its content. The route already exists in workspace-nav.tsx — no nav changes are needed. The page itself (`app/dashboard/page.tsx` or similar) should check `session.user.role` and render the appropriate role-specific dashboard component.

### Email Notification Trigger Points

| Trigger | Recipient | Template |
|---------|-----------|---------|
| Feedback transitions to "Under Review" | Feedback submitter | Feedback Under Review |
| Feedback transitions to Accepted/Rejected/Partially Accepted | Feedback submitter | Feedback Decision |
| New version published | All stakeholders assigned to any section in that policy | Version Published |
| User assigned to a section | Assigned user | Section Assignment |
| CR transitions to "Approved" | CR owner | CR Approved |
| Workshop created (Phase 10 — stub only in Phase 8) | Registered participants | Workshop Reminder |

### In-App Notification Generation

In-app notifications (NOTIF-01) should be generated by the same mutations that already exist in the tRPC routers (Phase 4 `transitionFeedback`, Phase 5 `transitionCR`, Phase 6 `publishVersion`, Phase 4 section assignment). The Phase 8 plan should add a `createNotification` service call at the end of each relevant mutation, writing to a new `notifications` table.

### "What Changed Since Last Visit" Data Strategy

The Stakeholder dashboard's "What changed" section queries for document versions published after `lastVisitAt` that contain sections the stakeholder is assigned to. The Policy Lead's "What changed" banner queries for feedback submitted after `lastVisitAt` across their managed policies. Both queries are straightforward joins on existing tables. No new schema beyond `users.lastVisitAt` and the `notifications` table.
