<objective>
Migrate all 7 `createNotification(...).catch(console.error)` callsites to `sendNotificationCreate(...)` awaited emits. This is the NOTIF-04 delivery — it moves notification work off the mutation critical path and onto Inngest's durable queue.

Per-file scope (from RESEARCH §Callsite Inventory lines 36-48):
- **feedback.ts** — 2 callsites: `startReview` (line ~343), `close` (line ~435). The `feedback.decide` callsite at line 398 is ALREADY on Inngest via `sendFeedbackReviewed` — do NOT touch it (it fires feedbackReviewedFn which has a richer payload than notification.create).
- **changeRequest.ts** — 3 callsites: `submitForReview` (line ~275), `approve` (line ~325), `merge` (line ~382).
- **version.ts** — 1 loop in `publish` (line ~149). The loop fans out `createNotification` per assigned user AND a second loop fans out `sendVersionPublishedEmail`. Both loops collapse into one loop of `sendNotificationCreate` per user (the new Inngest function handles the email inline with the DB insert).
- **sectionAssignment.ts** — 1 callsite in `assign` (line ~64) plus `sendSectionAssignedEmail` (line ~82). Both collapse into one `sendNotificationCreate` call.

NOTIF-06 dual-write safety: the legacy `createNotification` helper in `src/lib/notifications.ts` is NOT deleted. The `idempotency_key` unique partial index added in Plan 01 ensures that if both paths fire (e.g. during a deployment), only one row lands per (createdBy, entityType, entityId, action). Legacy callsites insert NULL idempotency_key → unaffected by the partial index.

Output: 4 edited router files. No new files. Zero changes to services, schema, or UI.
</objective>
