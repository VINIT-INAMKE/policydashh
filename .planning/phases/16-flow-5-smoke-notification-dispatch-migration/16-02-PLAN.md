<objective>
Build the durable dispatcher that takes `notification.create` events off the tRPC mutation critical path and handles both DB insert and email send with Inngest's retry + memoization semantics.

Purpose: Every mutation in feedback.ts, changeRequest.ts, version.ts, and sectionAssignment.ts currently does `createNotification(...).catch(console.error)` — a fire-and-forget path with no retry and no observability. Moving this into Inngest gives us: (a) 3 automatic retries on transient DB / Resend failures, (b) per-step timeline in the Inngest Dev UI, (c) dual-write idempotency via `onConflictDoNothing()` on the index added in Plan 01.

This plan creates the function and registers it. Plan 03 will migrate the callsites.

Output: A single new function file (`notification-dispatch.ts`), one-line register in the barrel (`functions/index.ts`), and a GREEN notification-dispatch.test.ts.
</objective>

    ```typescript
    import { NonRetriableError } from 'inngest'
    import { eq } from 'drizzle-orm'
    import { Resend } from 'resend'
    import { inngest } from '../client'
    import { notificationCreateEvent, computeNotificationIdempotencyKey } from '../events'
    import { db } from '@/src/db'
    import { notifications } from '@/src/db/schema/notifications'
    import { users } from '@/src/db/schema/users'

    const resend = process.env.RESEND_API_KEY
      ? new Resend(process.env.RESEND_API_KEY)
      : null

    const FROM_ADDRESS = process.env.RESEND_FROM_ADDRESS || 'PolicyDash <onboarding@resend.dev>'

    /**
     * notification-dispatch — off-critical-path notification + email dispatcher.
     *
     * Triggered by `notification.create` events emitted from tRPC mutations in
     * feedback.ts, changeRequest.ts, version.ts, sectionAssignment.ts. Handles
     * the DB insert with idempotency guard and dispatches a generic email for
     * any notification type with an email-capable user.
     *
     * NOTIF-05: replaces inline createNotification(...).catch(console.error).
     * NOTIF-06: onConflictDoNothing on idempotency_key makes dual-write safe.
     *
     * Phase 16 scope: email body is the notification's title + body text.
     * Type-specific rich email templates (e.g. CR status change) are deferred
     * to later phases. `feedback.reviewed` continues to use feedbackReviewedFn
     * which has a structured email template.
     */
    export const notificationDispatchFn = inngest.createFunction(
      {
        id: 'notification-dispatch',
        name: 'Notification dispatch — DB insert + email',
        retries: 3,
        triggers: [{ event: notificationCreateEvent }],
      },
      async ({ event, step }) => {
        const data = event.data

        // Step 1: insert notification row with idempotency guard.
        // Duplicate dispatches (dual-write window or Inngest retry after partial
        // execution) silently no-op at the DB level via onConflictDoNothing.
        await step.run('insert-notification', async () => {
          const idempotencyKey = computeNotificationIdempotencyKey({
            createdBy: data.createdBy,
            entityType: data.entityType,
            entityId: data.entityId,
            action: data.action,
          })

          await db
            .insert(notifications)
            .values({
              userId:         data.userId,
              type:           data.type,
              title:          data.title,
              body:           data.body ?? null,
              entityType:     data.entityType ?? null,
              entityId:       data.entityId ?? null,
              linkHref:       data.linkHref ?? null,
              idempotencyKey,
            })
            .onConflictDoNothing()
        })

        // Step 2: resolve the target user's email (may be null for phone-only users).
        const recipientEmail = await step.run('fetch-user-email', async () => {
          const [row] = await db
            .select({ email: users.email })
            .from(users)
            .where(eq(users.id, data.userId))
            .limit(1)
          if (!row) {
            // NonRetriableError: userId does not exist; no retry can fix that.
            throw new NonRetriableError(`user ${data.userId} not found`)
          }
          return row.email ?? null
        })

        // Step 3: send generic email. Skip silently when:
        //   - RESEND_API_KEY is unset (resend === null)
        //   - user has no email (phone-only)
        //   - notification type is cr_status_changed (in-app only per Phase 16 scope)
        const SKIP_EMAIL_TYPES = new Set(['cr_status_changed'])
        if (resend && recipientEmail && !SKIP_EMAIL_TYPES.has(data.type)) {
          await step.run('send-email', async () => {
            await resend!.emails.send({
              from:    FROM_ADDRESS,
              to:      recipientEmail,
              subject: data.title,
              text:    data.body ?? data.title,
            })
          })
        }

        return {
          userId: data.userId,
          type: data.type,
          action: data.action,
          emailed: Boolean(resend && recipientEmail && !SKIP_EMAIL_TYPES.has(data.type)),
        }
      },
    )
    ```

    CRITICAL RULES (from README.md + research pitfalls):
    - `triggers: [{ event: notificationCreateEvent }]` MUST be inlined inside the options literal. Do NOT extract to a const — Inngest v4 type widening footgun per README line 91-94.
    - Use `step.run` for every side effect; memoization gives us retry safety for free.
    - Throw `NonRetriableError` ONLY for missing user (deterministic failure). Transient DB / Resend errors should bubble as plain `Error` so Inngest retries.
    - Do NOT call `sendFeedbackReviewedEmail` / `sendVersionPublishedEmail` / `sendSectionAssignedEmail` from here — they require structured payloads not carried in notification.create. Phase 16 uses a generic title/body send. (Plan 04 verifies this keeps Flow 5 semantics intact because Flow 5 continues to use feedbackReviewedFn, NOT this new function.)
