## Surface Specifications

### 1. PolicyTabBar — "Milestones" tab addition

- New tab label: `"Milestones"` (matches existing title-case convention: Content, Feedback, Change Requests, Versions, Traceability)
- `href`: `/policies/${documentId}/milestones`
- `match`: `'startsWith'`
- `visible`: `canViewMilestones` (role is admin, moderator, or auditor)
- Insert position: between "Versions" and "Traceability" (reflecting lifecycle order)
- Layout prop added: `canViewMilestones: boolean` — passed from `layout.tsx` using the same DB query already present there
- No visual change to existing tabs; active underline treatment unchanged

### 2. Milestone Index Page — `/policies/[id]/milestones`

**Layout:** Server component. Full-width content area under tab bar. Two-column header row (title left, "Create milestone" button right), then `MilestoneList`.

**Page title:** `"Milestones"` — `text-xl font-semibold` at 20px/600

**Create CTA:** `Button` variant=`"default"` size=`"default"`, label `"Create milestone"`. Positioned top-right of page header. Opens `CreateMilestoneDialog`.

**MilestoneList:**
- Each row is a `Card` with `hover:shadow-sm` transition
- Row contents (left to right): milestone title (`text-sm font-medium`), description snippet truncated to 1 line (`text-sm text-muted-foreground line-clamp-1`), `MilestoneStatusBadge`, created date (`text-xs text-muted-foreground`), `ChevronRight` icon
- Click navigates to `/policies/[id]/milestones/[milestoneId]`
- Card padding: `p-4` (16px)
- Gap between cards: `space-y-2` (8px)

**Empty state (no milestones yet):**
- Centered vertically in content area
- Heading: `"No milestones yet"` — `text-sm font-medium`
- Body: `"Create a milestone to group versions, workshops, feedback, and evidence for Cardano anchoring."` — `text-sm text-muted-foreground`
- CTA: same "Create milestone" `Button` below the body text

### 3. Create Milestone Dialog

**Trigger:** "Create milestone" button on index page header
**Dialog size:** Standard shadcn Dialog — max-width `sm:max-w-md`
**Header:** Title `"Create milestone"`, description `"Define required slot counts then add entities on the detail page."`

**Form fields (top to bottom):**

| Field             | Element        | Validation              | Placeholder / Helper text                        |
|-------------------|----------------|-------------------------|--------------------------------------------------|
| Name              | `Input`        | required, min 1 char    | placeholder `"e.g. v1.0 Consultation Milestone"` |
| Description       | `Textarea`     | optional, rows=3        | placeholder `"Optional — what does this milestone represent?"` |
| Required versions | `Input` type=number | min 0, default 0   | label `"Versions"`, helper `"Minimum linked versions"`  |
| Required workshops| `Input` type=number | min 0, default 0   | label `"Workshops"`, helper `"Minimum linked workshops"` |
| Required feedback | `Input` type=number | min 0, default 0   | label `"Feedback items"`, helper `"Minimum linked feedback items"` |
| Required evidence | `Input` type=number | min 0, default 0   | label `"Evidence artifacts"`, helper `"Minimum linked artifacts"` |

**Slot input layout:** 2×2 grid (`grid grid-cols-2 gap-4`) for the 4 number inputs
**Footer:** Cancel button (variant=`"outline"`) + "Create milestone" submit button (variant=`"default"`), right-aligned
**Submit state:** Submit button shows `Loader2` spinner + `"Creating…"` label while mutation is pending; disabled during pending
**Success:** Dialog closes, router refreshes, user lands on new milestone detail page automatically

### 4. Milestone Detail Page — `/policies/[id]/milestones/[milestoneId]`

**Layout:** Client component (requires tRPC hooks). Full-width under tab bar. Stacked: `MilestoneDetailHeader` → `MilestoneDetailTabs`.

#### MilestoneDetailHeader

Structure (left to right on top row):
- Milestone title — `text-xl font-semibold` (20px/600)
- `MilestoneStatusBadge` — inline, 8px left margin

Second row (slot status indicators):
- One `MilestoneSlotStatus` per required slot type that has a non-zero requirement
- Slot types with `requiredSlots[type] === 0` (or absent) show no indicator — they are unconstrained
- Indicators displayed inline, `gap-3` between them

Third row:
- `"Mark ready"` button (variant=`"default"` size=`"default"`) — right-aligned via flex justify-between
- Disabled and `title="All required slots must be met"` when any non-zero slot is unmet
- Disabled and `title="Milestone is read-only"` when status is `ready`, `anchoring`, or `anchored`
- On click: fires `trpc.milestone.markReady` mutation; shows spinner during pending; on error renders `MarkReadyErrorDisplay` inline below the header

**Post-anchored read-only state:**
- "Mark ready" button hidden
- `MilestoneStatusBadge` shows `anchored` with `Lock` icon
- All checkboxes in `MilestoneEntityTab` disabled with `cursor-not-allowed`
- SHA256 hash displayed: `font-mono text-xs text-muted-foreground` in a `<code>` element below the slot status row, prefixed with label `"SHA256:"` — only when `contentHash` is non-null

#### MilestoneSlotStatus Component

One element per slot type. Inline `<span>` structure:

```
[entityType label]: [current]/[required] [icon]
```

Examples:
- Met: `"Versions: 2 / 1"` + `CheckCircle` lucide icon size-3.5 in emerald-600 (`text-[var(--status-cr-approved-text)]`)
- Unmet: `"Workshops: 0 / 1"` + `X` lucide icon size-3.5 in `text-destructive`

Typography: `text-sm` label + count, `font-medium` for the count portion only
Spacing: `inline-flex items-center gap-1`

#### MilestoneDetailTabs — 4 tabs

Use `Tabs` + `TabsList` + `TabsTrigger` + `TabsContent` from `components/ui/tabs.tsx` with `variant="line"` on `TabsList`.

Tab labels and order:
1. `"Versions"` — links to entity type `version`
2. `"Workshops"` — links to entity type `workshop`
3. `"Feedback"` — links to entity type `feedback`
4. `"Evidence"` — links to entity type `evidence`

Default active tab: `"Versions"`

Tab content area: `pt-4` (16px top padding), full width, `min-h-[240px]`

#### MilestoneEntityTab (content of each tab)

Each tab renders a list of entities of that type belonging to this policy. Two sections within the tab:

**Section A — "Attached to this milestone" (top)**
- Header: `"Attached"` label `text-xs font-medium text-muted-foreground uppercase tracking-wide`
- List of entities already linked (`milestoneId = this milestone`)
- Each row: `Checkbox` (checked, clicking removes) + entity display name + metadata (date / label)
- Checked checkbox: `aria-label="Remove [entity name] from milestone"`

**Section B — "Available to attach" (bottom)**
- Header: `"Available"` label (same style as above)
- List of entities belonging to this policy with `milestoneId IS NULL` (not yet in any milestone)
- Each row: `Checkbox` (unchecked, clicking attaches) + entity display name + metadata
- Unchecked checkbox: `aria-label="Add [entity name] to milestone"`

**Row layout:** `flex items-center gap-3 py-2 px-0` — no card wrapper, borderless rows with `border-b border-border` dividers
**Checkbox size:** 16px × 16px (`size-4`)
**Empty section:** `"No [entity type] attached yet."` in `text-sm text-muted-foreground` — shown in section A when nothing attached
**Skeleton loading:** 3× `Skeleton className="h-8 w-full"` while query loads

**Checkbox mutation behavior:**
- Optimistic UI: checkbox state flips immediately on click
- On error: checkbox state reverts + `toast.error(...)` (sonner, existing pattern)
- No confirmation dialog for attach/detach — reversible low-risk action

**Read-only state (status is `ready`, `anchoring`, or `anchored`):**
- All checkboxes `disabled` with `pointer-events-none opacity-50`
- Section header shows `"(read-only)"` suffix in `text-muted-foreground`

### 5. MarkReadyErrorDisplay

Shown inline in `MilestoneDetailHeader` when `markReady` mutation fails with unmet-slot error.

Layout: `rounded-md border border-destructive/30 bg-destructive/5 px-4 py-3 mt-3`

Content:
- Heading: `"Cannot mark ready — requirements not met"` (`text-sm font-medium text-destructive`)
- Unordered list of unmet slots, one per line: `"Workshops: 0 linked, 1 required"` (`text-sm text-muted-foreground`)
- Each list item prefixed with `X` lucide icon size-3.5 `text-destructive`

Dismissal: disappears when user attaches more entities or on next successful mutation

## Interaction Contract

### Checkbox attach/detach flow

1. User clicks unchecked checkbox on an "Available" entity row
2. Checkbox flips to checked immediately (optimistic)
3. `trpc.milestone.attachEntity` mutation fires with `{ milestoneId, entityType, entityId }`
4. On success: entity moves from "Available" section to "Attached" section; slot status indicator updates; "Mark ready" button re-evaluates enabled state
5. On error: checkbox reverts to unchecked; `toast.error("Failed to update milestone. Try again.")`

### Mark ready flow

1. "Mark ready" button is `disabled` until all required slots are met
2. On click: button shows `Loader2` spinner + `"Marking ready…"` label; button disabled
3. `trpc.milestone.markReady` mutation fires
4. On success: status badge changes to `"Ready"`; all entity tab checkboxes become disabled; "Mark ready" button hidden; `toast.success("Milestone marked ready")`
5. On error: button returns to enabled state; `MarkReadyErrorDisplay` renders below header with structured unmet-slot list

### Create milestone dialog flow

1. Admin clicks "Create milestone" on index page
2. Dialog opens; focus moves to Name input
3. Admin fills Name (required), optional Description, and slot counts (default 0)
4. "Create milestone" submit clicked
5. Button enters `"Creating…"` pending state
6. On success: dialog closes; `router.push(/policies/[id]/milestones/[newId])`
7. On error: inline error message below submit button: `"Failed to create milestone. Try again."` — dialog stays open
