# Phase 11 — UI Design Contract

> Visual and interaction contract for Real-Time Collaboration. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> Design system inherited from Phase 3 UI-SPEC. All tokens are continuations, not redeclarations.
> This phase adds a presence layer and comment system on top of the existing BlockEditor. No new routes.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (already initialized — `components.json` confirmed) |
| Style | base-nova |
| Preset | neutral base color, CSS variables enabled, Geist font |
| Component library | @base-ui/react (base-nova uses Base UI, not Radix) |
| Icon library | lucide-react |
| Font | `var(--font-geist-sans)` for all UI; `var(--font-geist-mono)` for code blocks and inline code |

**shadcn status:** `components.json` exists. Style is `base-nova`. All previously installed components available: alert-dialog, badge, button, card, checkbox, command, dialog, dropdown-menu, input, label, popover, progress, radio-group, scroll-area, select, separator, skeleton, sonner, table, tabs, textarea, tooltip, accordion, collapsible, sheet.

**New shadcn components needed for Phase 11:** None. All required UI primitives are already installed.

**New npm packages needed (not shadcn):**

| Package | Purpose | Safety |
|---------|---------|--------|
| `yjs` | CRDT document model for conflict-free collaborative editing | Official Yjs package — 15k+ GitHub stars, widely used |
| `@hocuspocus/provider` | WebSocket provider connecting Yjs to Hocuspocus server | Official Hocuspocus package by ueberdosis (Tiptap authors) |

**Hocuspocus server:** Runs as a separate Node.js process (not on Vercel). Required deployment target: Railway, Fly.io, or VPS. This is outside the Next.js application boundary. The UI-SPEC only covers client-side visual contracts.

## Color

Inherited from Phase 3. CSS custom properties from `app/globals.css` (oklch-based neutral palette). Extended with per-user presence colors.

| Role | Light Value | Dark Value | Usage |
|------|-------------|------------|-------|
| Dominant (60%) | `oklch(1 0 0)` (white) | `oklch(0.145 0 0)` (near-black) | Editor canvas background, comment panel background |
| Secondary (30%) | `oklch(0.97 0 0)` (neutral-50) | `oklch(0.269 0 0)` (neutral-800) | Comment thread background, presence bar background, resolved comment bg |
| Accent (10%) | `oklch(0.205 0 0)` (near-black) | `oklch(0.922 0 0)` (near-white) | See reserved list below |
| Destructive | `oklch(0.577 0.245 27.325)` (red) | `oklch(0.704 0.191 22.216)` (red) | No destructive actions in Phase 11 |

All color values referenced via CSS custom properties. Never hardcode oklch values directly.

**Accent reserved for these specific elements only (Phase 11 additions):**
1. "Post comment" primary submit button (filled) — `var(--primary)` background
2. Active comment thread anchor highlight (text selection background in editor) — `var(--primary)/15`
3. Connection status "Connected" dot — `var(--primary)` (reused as positive-state color)
4. (Inherited from Phase 3) Active/selected slash command menu item, active toolbar formatting buttons, drag handle hover, focused block outline, table cell selection

**Presence colors — user cursor and avatar colors:**

Each connected user is assigned a deterministic color from the palette below, derived from their user ID hash. Colors must pass 3:1 contrast ratio against white background for the cursor label text.

| Slot | Hex (cursor bg) | Text color |
|------|----------------|------------|
| 0 | `#7C3AED` (violet-600) | white |
| 1 | `#0891B2` (cyan-600) | white |
| 2 | `#059669` (emerald-600) | white |
| 3 | `#D97706` (amber-600) | white |
| 4 | `#DC2626` (red-600) | white |
| 5 | `#9333EA` (purple-600) | white |
| 6 | `#0284C7` (sky-600) | white |
| 7 | `#16A34A` (green-600) | white |

Assignment: `colorIndex = parseInt(userId.slice(-4), 16) % 8`. If overflow > 7 users, cycle modulo.

**Comment state colors:**

| State | Background | Border | Text |
|-------|------------|--------|------|
| Open comment thread | `var(--background)` | `var(--border)` | `var(--foreground)` |
| Resolved comment thread | `var(--muted)` | `var(--border)` | `var(--muted-foreground)` |
| Comment anchor (highlighted text) | `var(--primary)/15` | none | `var(--foreground)` (unchanged) |
| Focused/selected comment anchor | `var(--primary)/25` | none | `var(--foreground)` |

**Connection status semantic colors:**

| State | Indicator color | CSS |
|-------|----------------|-----|
| Connected | Green dot | `#16A34A` (green-600) — hardcoded; semantic meaning requires specific green |
| Reconnecting | Amber dot with pulse | `#D97706` (amber-600) — hardcoded |
| Offline | Red dot | `#DC2626` (red-600) — hardcoded |

Connection status dot is 8px diameter circle. These three colors are the only hardcoded values in this phase; all others use CSS variables.

**Do NOT use presence colors for:** any non-presence UI element. Presence colors are strictly for cursor backgrounds, cursor labels, and presence avatars. They must not bleed into comment panels, buttons, or page chrome.

## Component Inventory

### Layout Change from Phase 3

The existing policy section editor layout is extended as follows:

```
[EditorHeader]
  [Left: Section title] [Right: PresenceBar + ConnectionStatus + existing Save controls]

[EditorContent — existing BlockEditor]
  [Remote cursors rendered inline by @tiptap/extension-collaboration-cursor]
  [Comment anchors — highlighted text ranges with CommentBubble trigger on margin]

[CommentPanel — 320px right side panel, slides in/out]
  [CommentPanelHeader]
  [CommentList]
    [CommentThread(s)]
```

The `CommentPanel` is conditionally rendered. When collapsed: editor uses full available width. When expanded: editor content max-width 768px is maintained; the panel appears to the right with its own 320px allocation (total page width widens, sidebar + editor + comments).

On viewports < 1200px wide: `CommentPanel` renders as a Sheet (slide-in from right, overlays editor) rather than inline panel.

### Component: PresenceBar

Displayed in the editor header, right side, before the connection status and save indicator.

Shows avatars of all users currently viewing or editing this section (including the current user).

| Element | Spec |
|---------|------|
| Container | Flex row, gap-xs (4px), align-center |
| Avatar | 28px circle, bg = presence color for that user, text = 10px/600 white initials (first letter of first name + first letter of last name, e.g. "AK"). border-2 border-background (creates stacking gap). |
| Avatar stack | Negative margin-left -8px on each avatar after the first; `z-index` increases left-to-right so leftmost is on top |
| Avatar tooltip | Tooltip on hover: full display name + "(you)" suffix for own avatar. Delay: 300ms. |
| Overflow pill | If > 5 users: show first 4 avatars + "+N" pill. "+N" pill: same 28px circle, bg-muted, text 10px/600 foreground. Tooltip: lists all overflow names. |
| Empty state | No avatars shown when only current user is present (single-user editing). PresenceBar is hidden at 0 avatars width. |

**Avatar initials derivation:** Split `displayName` by spaces. Take `name[0][0].toUpperCase()` + `name[name.length-1][0].toUpperCase()`. If single name, use first two chars. Fallback: "?".

Displayed in the editor header, right side, after PresenceBar and before auto-save indicator.

| Element | Spec |
|---------|------|

### Component: RemoteCursor

Rendered by `@tiptap/extension-collaboration-cursor`. The extension injects cursor markup into the ProseMirror document. Visual contract for the custom cursor render function:

| Element | Spec |
|---------|------|
| Cursor caret | 2px wide vertical line. Color: user's presence color. Height: line-height of surrounding text. |
| Cursor label | Positioned above the caret. Pill shape: 4px border-radius top-left, top-right, bottom-right; 0px bottom-left (points to caret). Padding: 2px 6px. Background: user's presence color. Text: 12px/600 white, user's display name. max-width 120px, truncate with ellipsis. |
| Label visibility | Always visible when cursor is active. Fades to opacity-0 after 3s of inactivity on that cursor, returns on movement. `transition-opacity 300ms`. |
| Own cursor | Not rendered (standard browser caret is used for the local user). |
| Z-index | All remote cursors at z-50. Labels at z-51. |

### Component: CommentAnchor

An inline text decoration rendered on comment-annotated text ranges. Not a React component — implemented as a ProseMirror decoration or Tiptap mark.

| Element | Spec |
|---------|------|
| Highlighted text | Background: `var(--primary)/15`. No border-bottom or underline (avoids conflict with link marks). |
| Active anchor (comment panel focused on this thread) | Background: `var(--primary)/25`. |
| Resolved anchor | No highlight. Decoration removed when comment thread is resolved. |
| Comment bubble trigger | Small icon button positioned at the right margin of the highlighted range's last line. Size: 20px. Icon: `MessageSquare` (lucide), 12px, muted-foreground. Background: `var(--background)`, border `var(--border)`, rounded-full. On hover: bg-muted. On click: opens CommentPanel scrolled to the relevant thread. |
| Bubble trigger visibility | Hidden by default. Visible on hover of the highlighted text range. Always visible when CommentPanel is open and that thread is active. |

### Component: CommentPanel

Right-side panel (320px fixed) showing all comment threads for the current section, grouped by open/resolved.

**Panel structure:**

```
[CommentPanelHeader]
  "Comments" (Heading 20px/600) + close button (X icon, ghost, 32px)
  [Tab row: "Open ({N})" | "Resolved ({N})"]

[CommentPanelBody — ScrollArea, fills remaining height]
  [NewCommentForm — shown when creating a new comment]
  [CommentThreadList]
    [CommentThread × N]

[CommentPanelEmpty — shown when no threads in active tab]
```

**Tabs:** Use `Tabs` shadcn component. "Open" tab is default. Count badges on each tab label showing thread count.

### Component: CommentThread

A single comment thread: one root comment + N replies.

| Element | Spec |
|---------|------|
| Thread container | bg-background, border border-border, rounded-md, padding md (16px). margin-bottom sm (8px). |
| Thread header | Row: author avatar (20px circle, presence color if online, muted otherwise) + author name (12px/600) + timestamp (12px/400, muted-foreground). Right side: thread action menu (`DropdownMenu`, MoreHorizontal trigger, 24px). |
| Root comment body | 14px/400/1.5. Full text, no truncation. mt-4px below header. |
| Reply list | mt-sm (8px) margin-top. Each reply indented 16px left with `border-l-2 border-border` visual. |
| Reply item | Author name (12px/600) + timestamp (12px/400) row. Body (14px/400/1.5) below. |
| Reply form | Shown inline below replies when "Reply" is clicked. Same layout as NewCommentForm but compact. Collapse on Escape or clicking away without text. |
| "Reply" button | `Button` ghost size="sm", 12px, muted-foreground. Located below last reply. |
| "Resolve" button | `Button` outline size="sm" — "Resolve". Located in thread action menu AND as a standalone button at thread bottom for open threads. Icon: `CheckCheck` (lucide), 14px. |
| "Reopen" button | `Button` outline size="sm" — "Reopen". Shown for resolved threads. Icon: `RotateCcw` (lucide), 14px. |
| Resolved state | Thread container bg-muted, all text muted-foreground, "Resolved" badge (Badge secondary, 12px) next to thread header. Reply form hidden. |

**Thread action menu items (DropdownMenu):**

For open thread owned by current user:
- "Resolve thread" (with CheckCheck icon)
- separator
- "Delete comment" (destructive text color, no separate confirm — toast undo pattern)

For open thread not owned by current user (moderator/admin):
- "Resolve thread"

For resolved thread:
- "Reopen thread"

### Existing Component: BlockEditor (modifications for Phase 11)

The `BlockEditor` component is extended. Changes are additive — no existing props or behaviors are removed.

| Change | Spec |
|--------|------|

## Interaction States

### Presence States

| State | Visual Treatment |
|-------|-----------------|
| User joins session | Avatar slides in from right in PresenceBar with 150ms ease-out. |
| User leaves session | Avatar fades out 300ms ease-out, stack reflows. |
| User cursor active | Cursor label fully opaque. |
| User cursor idle (3s) | Cursor label fades to opacity-0 (transition 300ms). Cursor caret remains. |
| User cursor moves | Label fades back to opacity-100 instantly. |

### Comment Anchor States

| State | Visual Treatment |
|-------|-----------------|
| Default (text has comment) | `var(--primary)/15` background on highlighted range. |
| Hover on highlighted text | CommentBubble trigger appears at margin. Background stays same. |
| Active (panel focused on this thread) | `var(--primary)/25` background. |
| Resolved | No highlight. Decoration removed. |

### Comment Thread States

| State | Visual Treatment |
|-------|-----------------|
| Open | bg-background, full opacity, reply + resolve available. |
| Resolved | bg-muted, all text muted-foreground/70, "Resolved" badge visible, reply hidden, reopen button shown. |
| Loading (submitting reply) | Submit button shows Loader2 spinner, disabled. Textarea disabled. |
| Error | Submit button re-enables. Error toast shown. Input value retained. |

### CommentPanel States

| State | Visual Treatment |
|-------|-----------------|
| Closed | Panel hidden. Editor uses full available width. |
| Open (inline, lg+) | Panel slides in from right (translateX 0 from translateX(100%), 200ms ease-out). Editor remains at max-width 768px. |
| Open (sheet, < 1200px) | Uses `Sheet` component (from right). Overlays editor. |
| Loading (initial fetch) | 3 Skeleton comment threads (h-20 w-full) with sm gaps. |

### Connection States

| State | Visual Treatment |
|-------|-----------------|
| Connected | Green dot. Editor fully interactive. |
| Reconnecting | Amber pulsing dot. Editor still editable (optimistic). Toast banner at top of editor: "Reconnecting to collaboration server…" — non-dismissible, auto-hides when reconnected. |
| Offline (max retry exceeded) | Red dot. "Offline" label visible. Toast: "Connection lost. Changes will be saved locally." Editor remains editable — auto-save fallback activates. |
| Reconnected after offline | Green dot restored. "Offline" toast dismissed. Sync occurs transparently. Toast: "Reconnected. Changes synced." |

## Accessibility Requirements

| Requirement | Implementation |
|-------------|----------------|
| Keyboard navigation | CommentBubble accessible via `Ctrl+Alt+M` keyboard shortcut. CommentPanel focusable via Tab. CommentThread items navigable with Tab. Resolve/Reopen buttons reachable. Reply form accessible via keyboard. |
| Focus management | When CommentPanel opens (from CommentBubble click), focus moves to the `Textarea` in NewCommentForm. When CommentPanel closes, focus returns to editor at original cursor position. |
| Screen reader | PresenceBar: `aria-label="N users viewing this section"` on container. Each avatar: `aria-label="{Name} is viewing this section"`. ConnectionStatus: `role="status"` + `aria-live="polite"` (announces state changes). CommentBubble button: `aria-label="Add comment to selected text"`. CommentThread resolve button: `aria-label="Resolve comment thread by {author}"`. Remote cursor carets: rendered as decorations, not in document flow — `aria-hidden="true"` on cursor DOM elements. |
| Color contrast | Presence colors (violet-600, cyan-600, emerald-600, amber-600, etc.) all pass 3:1 against white canvas background. White text on presence color backgrounds passes 4.5:1. Connection status dots pass 3:1. |
| Motion | Cursor label fade, avatar enter/exit, panel slide-in all respect `prefers-reduced-motion`: disable transitions, use instant show/hide. |
| Comment anchor | Comment anchor highlights do not rely on color alone — when CommentPanel is open, active thread anchor is also indicated by the visible `CommentBubble` trigger icon at the margin. |

## Phase-Specific Notes

### Integration Point: BlockEditor

The existing `BlockEditor` at `app/(workspace)/policies/[id]/_components/block-editor.tsx` is modified. The changes are:

1. `buildExtensions()` in `src/lib/tiptap-extensions/build-extensions.ts` receives a `collaboration` option. When `true`, the StarterKit's `history` is disabled and `@tiptap/extension-collaboration` + `@tiptap/extension-collaboration-cursor` are added instead.
2. `HocuspocusProvider` is instantiated in `BlockEditor` keyed on `sectionId`. The Yjs `Doc` is passed to the collaboration extensions.
3. `immediatelyRender: false` is already set — no change needed.
4. The existing auto-save debounce is gated on `!isCollaborating` to prevent double-writes when Hocuspocus is handling persistence.

### Comment System Architecture

Inline comments are anchored to text using a ProseMirror decoration approach (not a Tiptap mark stored in document JSON). This preserves content portability — comment anchors are stored separately in the database linked to character offsets or node IDs, not embedded in the document JSON. This avoids contaminating the Tiptap content with comment metadata that does not belong in the versioned document snapshot.

Comment data model (for executor reference):
- `comments` table: `id`, `sectionId`, `authorId`, `body`, `createdAt`, `resolvedAt`, `resolvedBy`, `anchorStart`, `anchorEnd` (character offsets in the current document)
- `commentReplies` table: `id`, `commentId`, `authorId`, `body`, `createdAt`

tRPC router: `commentRouter` with procedures `create`, `list`, `resolve`, `reopen`, `delete`, `createReply`.

### Hocuspocus Server

The Hocuspocus server is a separate deployment. Environment variable `NEXT_PUBLIC_HOCUSPOCUS_URL` must be set to the WebSocket endpoint (e.g., `wss://collab.policydash.example.com`). The `HocuspocusProvider` uses this URL. The client handles connection errors gracefully with the connection status UI defined above.

### Auto-Save Fallback

When `HocuspocusProvider` connection state is `disconnected` (not `connected` or `connecting`):
- Auto-save via `updateSectionContent` tRPC mutation activates with the existing 1.5s debounce
- ConnectionStatus shows "Offline" state
- On reconnect, Hocuspocus performs CRDT merge — no data loss

No new routes. All collaboration UI layers onto the existing `/policies/[id]` page.
