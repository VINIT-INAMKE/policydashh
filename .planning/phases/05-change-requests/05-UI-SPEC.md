## Spacing Scale

Inherited from Phase 2 and Phase 4. No changes.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding, CR-ID badge internal padding |
| sm | 8px | Compact element spacing, card internal gaps, form field gaps |
| md | 16px | Default element spacing, card padding, list item padding |
| lg | 24px | Section padding, dialog body padding, group spacing |
| xl | 32px | Layout column gaps, page section margins |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level top/bottom spacing |

**Phase 5 exceptions:**

| Element | Value | Reason |
|---------|-------|--------|
| CR-ID badge width | min-content, no truncation | CR-001 must always be fully readable |
| CR detail linked feedback list max-height | 320px with scroll | Prevents runaway layout on CRs with many linked feedback items |
| Merge summary textarea min-height | 96px (6 rows) | Significant action; discourage one-liners |
| Closure rationale textarea min-height | 96px (6 rows) | Mandatory field; same weight as feedback rationale |
| CR detail page content max-width | 800px | Wider than feedback detail to accommodate section links table |

**Source:** Inherited from Phase 2 UI-SPEC (spacing), Phase 4 UI-SPEC (textarea min-height pattern)

## Typography

Inherited from Phase 2 and Phase 4. No changes to the four declared roles.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (regular) | 1.5 | CR descriptions, linked feedback body preview, rationale text, section link labels |
| Label | 12px | 400 (regular) | 1.4 | Form labels, timestamps, metadata (status, owner, date), CR-ID badges, section reference IDs |
| Heading | 20px | 600 (semibold) | 1.2 | Page titles, dialog titles, CR detail title |
| Display | 28px | 600 (semibold) | 1.2 | Not used in Phase 5 (reserved for document titles in Phase 2) |

Font family: `var(--font-geist-sans)` for all UI. `var(--font-geist-mono)` for human-readable IDs (CR-001, FB-001) displayed in list views, detail views, and merge confirmations.

**Declared weights: 400 (regular) and 600 (semibold) only.** No additional weights introduced in this phase.

**CR ID display:** Render CR-NNN IDs in `var(--font-geist-mono)` at 12px/400 inside a muted badge — identical treatment to FB-NNN IDs in Phase 4. Both IDs may appear together on CR detail pages; consistent monospace treatment ensures visual differentiation from prose.

**Source:** Phase 2 UI-SPEC (typography), Phase 4 UI-SPEC (ID badge treatment)

## Color

Inherited from Phase 2 and Phase 4. CSS custom properties from `app/globals.css` (oklch-based neutral palette).

| Role | Light Value | Dark Value | Usage |
|------|-------------|------------|-------|
| Dominant (60%) | `oklch(1 0 0)` (white) | `oklch(0.145 0 0)` (near-black) | Page background, CR detail view, form backgrounds |
| Secondary (30%) | `oklch(0.97 0 0)` (neutral-50) | `oklch(0.269 0 0)` (neutral-800) | CR list background, filter sidebar background, metadata rows, linked feedback containers, linked sections table |
| Accent (10%) | `oklch(0.205 0 0)` (near-black) | `oklch(0.922 0 0)` (near-white) | See reserved list below |
| Destructive | `oklch(0.577 0.245 27.325)` (red) | `oklch(0.704 0.191 22.216)` (red) | "Close without Merge" action button, destructive confirmation dialogs |

All color values referenced via CSS custom properties only. Never hardcode oklch values directly.

**Accent reserved for these specific elements only:**
1. "Create Change Request" primary CTA button (filled)
2. "Approve" CR lifecycle action button (filled) — positive decision
3. "Merge" CR lifecycle action button (filled) — terminal positive action
4. Active filter chip/tag in CR list filter panel
5. CR status badge for "In Review" state (`bg-primary/10`, `text-primary`)
6. Focused form field ring (inherited from base-nova ring behavior)

**Do NOT use accent for:** status badges for Drafting/Approved/Merged/Closed states (use semantic colors below), secondary actions (use outline variant), navigation items, linked feedback badges, linked section references.

**Additional semantic colors for CR lifecycle:**

| Semantic | CSS Variable / Value | Usage |
|----------|---------------------|-------|
| Status: Drafting | `var(--muted)` bg + `var(--muted-foreground)` text | Neutral badge; CR is being authored |
| Status: In Review | `var(--primary)/10` bg + `var(--primary)` text | Accent-tinted badge (see reserved list) |
| Status: Approved | `oklch(0.9 0.08 145)` bg + `oklch(0.4 0.12 145)` text | Green badge; approved and ready to merge |
| Status: Merged | `oklch(0.88 0.07 265)` bg + `oklch(0.4 0.12 265)` text | Purple/indigo badge; permanently merged |
| Status: Closed | `var(--muted)` bg + `var(--muted-foreground)` text | Neutral badge; closed without merge |
| CR-ID badge | `var(--muted)` bg + `var(--muted-foreground)` text | Same as FB-NNN treatment in Phase 4 |

**Implementation note:** Declare `--status-cr-approved-bg`, `--status-cr-approved-text`, `--status-cr-merged-bg`, `--status-cr-merged-text` as custom CSS variables in `app/globals.css`. Do not hardcode oklch values at component call sites.

**Source:** Phase 4 UI-SPEC (status badge color pattern); merged status uses purple/indigo to distinguish from accepted feedback (green) — merged is a terminal state with special visual weight.

| Element | Component | Variant | Notes |

**Trigger:** "Create Change Request" button on the CR list page.

**Layout:** Dialog, max-width 560px. Two-step form: Step 1 — select source feedback items; Step 2 — fill CR metadata (title, description, owner).

**Step 1 — Select Source Feedback:**

| Element | Component | Variant | Notes |
|---------|-----------|---------|-------|
| Dialog | `Dialog` | -- | max-width 560px |

| Breakpoint | Behavior |

## Phase-Specific Notes

### What This Phase Does NOT Include

- **Version history UI**: Merging creates a new version record in the DB (CR-06), but the versioning UI (diff view, version archive, changelog page) ships in Phase 6. The MergeDialog shows the "New version will be created: v0.N" preview text only.
- **Traceability matrix**: The full feedback→CR→section→version chain view ships in Phase 7. CR detail shows direct links only.
- **Feedback status auto-update UI**: CR-07 states linked feedback is auto-updated on merge. This is a server-side operation only; no separate UI confirmation is needed.
- **Two-person approval**: WF-03 (two-person approval for CR merges) is a v2 requirement — not in this phase.
- **CR editing after creation**: Title, description, and owner are set at creation. Editing them post-creation is out of scope for this phase.

### CR Lifecycle State Machine Reference

```
Drafting → In Review (Submit for Review)
In Review → Approved (Approve)
In Review → Closed (Close without Merge + rationale)
Approved → Merged (Merge + merge summary)
Approved → Closed (Close without Merge + rationale)
```

Terminal states: Merged, Closed (no further transitions).

### ID Consistency Pattern

Both CR-NNN and FB-NNN IDs appear on the CR detail page. Both use:
- `var(--font-geist-mono)` typeface
- 12px / weight 400
- `var(--muted)` background + `var(--muted-foreground)` text
- Badge component (secondary variant)
- `aria-label` with full human-readable text

This consistency ensures the traceability chain is visually scannable when both ID types appear together.
