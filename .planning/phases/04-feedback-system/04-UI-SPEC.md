# Phase 4 — UI Design Contract

> Visual and interaction contract for the Feedback System phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> Design system inherited from Phase 2 and Phase 3 UI-SPECs. All tokens are continuations, not redeclarations.

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (already initialized — `components.json` confirmed) |
| Style | base-nova |
| Preset | neutral base color, CSS variables enabled, Geist font |
| Component library | @base-ui/react (base-nova uses Base UI, not Radix) |

**shadcn status:** `components.json` exists. Style is `base-nova`. Currently installed components: alert-dialog, badge, button, card, command, dialog, dropdown-menu, input, input-group, label, popover, scroll-area, separator, skeleton, sonner, table, tabs, textarea, tooltip.

| Token | Value | Usage |
|-------|-------|-------|

| sm | 8px | Compact element spacing, form field gaps, filter chip gaps |

| Element | Value | Reason |
|---------|-------|--------|

## Typography

Inherited from Phase 2 and Phase 3. No changes to the four declared roles.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (regular) | 1.5 | Feedback body text, list descriptions, rationale text |
| Label | 12px | 400 (regular) | 1.4 | Form labels, timestamps, metadata (status, priority, org type), feedback ID badges |
| Heading | 20px | 600 (semibold) | 1.2 | Page titles, section headings, dialog titles, feedback detail title |
| Display | 28px | 600 (semibold) | 1.2 | Not used in Phase 4 (reserved for document titles) |

Font family: `var(--font-geist-sans)` for all UI. `var(--font-geist-mono)` for human-readable IDs (FB-001, FB-002) displayed in detail views and audit contexts.

**Declared weights: 400 (regular) and 600 (semibold) only.** No additional weights introduced in this phase.

**Feedback ID display:** Render FB-NNN IDs in `var(--font-geist-mono)` at 12px/400 inside a muted badge to visually differentiate them from prose content.

## Color

Inherited from Phase 2 and Phase 3. CSS custom properties from `app/globals.css` (oklch-based neutral palette).

| Role | Light Value | Dark Value | Usage |
|------|-------------|------------|-------|
| Dominant (60%) | `oklch(1 0 0)` (white) | `oklch(0.145 0 0)` (near-black) | Page background, feedback detail view, form backgrounds |
| Secondary (30%) | `oklch(0.97 0 0)` (neutral-50) | `oklch(0.269 0 0)` (neutral-800) | Feedback inbox list background, filter sidebar background, metadata rows, evidence attachment containers |
| Accent (10%) | `oklch(0.205 0 0)` (near-black) | `oklch(0.922 0 0)` (near-white) | See reserved list below |
| Destructive | `oklch(0.577 0.245 27.325)` (red) | `oklch(0.704 0.191 22.216)` (red) | Reject action button, reject decision badge |

All color values referenced via CSS custom properties only. Never hardcode oklch values directly.

**Accent reserved for these specific elements only:**
1. "Submit Feedback" primary CTA button (filled)
2. "Accept" triage action button (filled) — positive decision
3. Active filter chip/tag in feedback inbox filter panel
4. Feedback status badge for "Under Review" state (bg-primary/10, text-primary)
5. Active section highlight in section-scoped feedback list (left border 2px primary)
6. Focused form field ring (inherited from base-nova ring behavior)

**Do NOT use accent for:** status badges for Submitted/Accepted/Rejected/Closed (use semantic colors below), secondary actions (use outline variant), navigation items, evidence attachment UI, filter labels.

**Additional semantic colors for feedback:**

| Semantic | CSS Variable | Usage |
|----------|-------------|-------|
| Status: Submitted | `var(--muted)` bg + `var(--muted-foreground)` text | Neutral badge for newly submitted feedback |
| Status: Under Review | `var(--primary)/10` bg + `var(--primary)` text | Accent-tinted badge (see reserved list) |
| Status: Accepted | `oklch(0.9 0.08 145)` bg (green-100 equivalent) + `oklch(0.4 0.12 145)` text | Custom green for accepted outcome |
| Status: Partially Accepted | `oklch(0.92 0.07 85)` bg (amber-100 equivalent) + `oklch(0.5 0.1 85)` text | Custom amber for partial outcome |
| Status: Rejected | `oklch(0.95 0.04 27)` bg (red-50 equivalent) + `var(--destructive)` text | Destructive-tinted badge for rejection |
| Status: Closed | `var(--muted)` bg + `var(--muted-foreground)` text | Same as Submitted; no action pending |
| Priority: High | `oklch(0.95 0.04 27)` bg + `var(--destructive)` text | Red-tinted priority indicator |
| Priority: Medium | `oklch(0.92 0.07 85)` bg + `oklch(0.5 0.1 85)` text | Amber-tinted priority indicator |
| Priority: Low | `var(--muted)` bg + `var(--muted-foreground)` text | Neutral priority indicator |
| Anonymous attribution | `var(--muted)` bg + `var(--muted-foreground)` text | Badge: "Anonymous" replaces stakeholder name |
| Evidence attached | `var(--muted-foreground)` icon (Paperclip, 12px) | Small inline indicator on feedback cards |
| Border | `var(--border)` | Card borders, separator lines, input borders |
| Muted foreground | `var(--muted-foreground)` | Secondary text, timestamps, placeholder text |
| Ring | `var(--ring)` | Focus rings on all interactive elements |

**Implementation note for status/priority semantic colors:** Declare these as inline Tailwind classes using arbitrary values or extend `app/globals.css` with custom CSS variables `--status-accepted-bg`, `--status-accepted-text`, etc. Do not hardcode oklch values at component call sites.

## Component Inventory

### Page: Feedback Inbox (`/policies/[id]/feedback`)

**Layout:** Two-column. Left: filter panel (240px fixed, border-r, bg-muted). Right: feedback list (flex-1). On mobile (<768px): filter panel collapses into a Sheet triggered by "Filters" button.

**Focal point declaration:** The eye should land first on the feedback count heading ("12 items") and the list of feedback cards below it. The filter panel is secondary — accessed on demand. Each feedback card's focal point is the title (14px/600 exception for card titles — see note below), then the status badge, then the FB-NNN ID.

**Note on card title weight:** Feedback card titles use 14px/600 (semibold body) to create scannable hierarchy in a dense list. This is the same pattern as Phase 2 policy card titles.

| Element | Component | Variant | Notes |
|---------|-----------|---------|-------|
| Page heading | `<h1>` | Heading (20px/600) | "Feedback — {section title}" |
| Item count | `<span>` | Label (12px/400), muted-foreground | "{N} items" beside heading |
| Filter panel heading | `<h2>` | Label (12px/400), uppercase tracking-wide | "FILTERS" |
| Clear filters button | `Button` | ghost, size="sm" | "Clear all" — only shown when filters active |
| Filter: Section | `Select` | -- | Label: "Section", lists all sections in the document |
| Filter: Org Type | Checkbox group | -- | Government, Industry, Legal, Academia, Civil Society, Internal |
| Filter: Priority | Checkbox group | -- | High, Medium, Low |
| Filter: Status | Checkbox group | -- | Submitted, Under Review, Accepted, Partially Accepted, Rejected, Closed |
| Filter: Type | Checkbox group | -- | Issue, Suggestion, Endorsement, Evidence, Question |
| Filter: Impact | Checkbox group | -- | Legal, Security, Tax, Consumer, Innovation, Clarity, Governance, Other |
| Feedback card | `Card` | default | See card spec below |
| Empty state | Custom div | -- | Centered, MessageSquare icon (48px, muted-foreground), heading + body copy |
| Loading state | `Skeleton` | -- | 5 skeleton cards matching card dimensions |
| Mobile filter trigger | `Button` | outline, size="sm" | Filter icon + "Filters" + active filter count badge |

**Feedback card layout:**

```
[CardHeader]
  Row: [Type badge] [Priority badge] [FB-NNN badge]      [Status badge (right)]
  Title (14px/600, truncated 1 line)
  Section name (12px/400, muted-foreground) — if inbox shows cross-section view
[CardBody]
  Body preview (14px/400, truncated 2 lines, muted-foreground)
[CardFooter]
  Row: [Org type] [Anonymous badge if anon]    [Paperclip icon if evidence] [Date (12px, muted-foreground)]
```

**Card click behavior:** Opens the Feedback Detail Sheet (slides in from right). Does not navigate away from the inbox.

### Component: TriageActions

Rendered at the bottom of the FeedbackDetail sheet. Visible only to users with Policy Lead or Admin role. Renders based on current XState feedback lifecycle state.

**State-based rendering:**

| Current Status | Available Actions |
|----------------|------------------|
| Submitted | "Mark Under Review" (outline button) |
| Under Review | "Accept" (default), "Partially Accept" (outline), "Reject" (destructive outline) |
| Accepted / Partially Accepted / Rejected | "Close" (outline) |
| Closed | No actions. Static text: "This feedback is closed." |

| Element | Component | Variant | Notes |
|---------|-----------|---------|-------|
| "Mark Under Review" | `Button` | outline, size="sm" | EyeIcon + "Mark Under Review" |
| "Accept" | `Button` | default, size="sm" | CheckIcon + "Accept" — opens rationale dialog |
| "Partially Accept" | `Button` | outline, size="sm" | CircleDashedIcon + "Partially Accept" — opens rationale dialog |
| "Reject" | `Button` | outline with destructive text, size="sm" | XIcon + "Reject" — opens rationale dialog |
| "Close" | `Button` | outline, size="sm" | ArchiveIcon + "Close" |

**"Mark Under Review" and "Close" transitions:** Fire immediately on click with a loading state. No rationale required.

**"Accept", "Partially Accept", "Reject" transitions:** Open the RationaleDialog before firing the mutation.

## Phase-Specific Notes

### XState Feedback Lifecycle

The feedback lifecycle is modeled as an XState 5 state machine. The machine is defined server-side (not in the React component tree) to ensure transitions are always validated before the database write.

**States and valid transitions:**

```
Submitted
  → Under Review (action: markUnderReview, no rationale required)

Under Review
  → Accepted (action: accept, rationale required)
  → Partially Accepted (action: partiallyAccept, rationale required)
  → Rejected (action: reject, rationale required)

Accepted | Partially Accepted | Rejected
  → Closed (action: close, no rationale required)

Closed
  (terminal — no transitions)
```

The tRPC mutation for each transition must:
1. Verify the current state from the database
2. Validate the transition is legal per the state machine
3. Persist the new state + decision log entry atomically
4. Write an audit log entry via `writeAuditLog()`

### Human-Readable ID Generation

FB-NNN IDs are generated at submission time. Implementation: a `feedback_sequence` counter per workspace, or a `ROW_NUMBER()` window function over `created_at` ordered rows per workspace. IDs are zero-padded to 3 digits minimum (FB-001 through FB-999), then expand naturally (FB-1000+). IDs are immutable once assigned.

### Section-Level Scoping (AUTH-05)

The feedback submission route `/policies/[id]/sections/[sectionId]/feedback/new` is gated by the stakeholder's section assignment. The tRPC `feedback:create` procedure checks `section_assignments` to verify the authenticated user is assigned to the requested section. Unassigned sections return 403. The stakeholder's section list in the nav only shows assigned sections.

The Policy Lead feedback inbox at `/policies/[id]/feedback` shows feedback from all sections in the document — no section scoping for Policy Lead (they manage all feedback on their documents).

### Privacy Architecture (AUTH-08)

Anonymous feedback is enforced server-side. When a feedback item is submitted with `anonymous: true`:
- The `submitter_id` is stored in the database (for audit purposes and for the stakeholder's own outcomes view)
- Any tRPC query result for this feedback item that goes to a non-owner (i.e., Policy Lead view) strips `submitter_id` and `submitter_name` from the response
- The client receives `{ submittedBy: null, isAnonymous: true }` for Policy Lead views
- The submitter sees their own name in the outcomes view (`/feedback/outcomes`) regardless of anonymity setting

### Section Content View Integration

In Phase 3, the policy detail page already has the section sidebar. In Phase 4:
- Add a "Submit Feedback" button to the section content area header (visible to Stakeholders with section assignment)
- Add a "Feedback ({N})" counter/tab to the section content area header (visible to Policy Leads)
- Do NOT embed the feedback inbox inside the block editor view; feedback is a separate route

### What This Phase Does NOT Include

- **Change Request creation**: CR workflow is Phase 5. Triage actions (Accept/Reject/Partial) in Phase 4 do NOT create CRs — that linkage is Phase 5.
- **Full traceability views**: TRACE-01 through TRACE-06 are Phase 7.
- **Dashboard integration**: UX-02 (Policy Lead dashboard feedback inbox) is Phase 8. Phase 4 ships the standalone feedback inbox page.
- **Email notifications**: NOTIF-02 is Phase 8. Phase 4 only shows in-app success toasts.
- **Search across feedback content**: SRCH-01/SRCH-02 are Phase 7. Phase 4 has filter-panel filtering only (exact match facets, not full-text search).
