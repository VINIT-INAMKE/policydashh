## Summary

Phase 20.5 ships two public server-rendered pages — `/research` (static content + PDF download) and `/framework` (dynamic draft consultation surface) — alongside three supporting artifacts: a Drizzle schema migration, a tRPC mutation for toggling `isPublicDraft`, and a service helper for the privacy-compliant what-changed log.

The codebase pattern is entirely established. Phase 9 already defined the `(public)` route group, `force-dynamic` SSR pattern, UUID guard, and PUB-05 privacy rules. Phase 19 established the proxy.ts append pattern. The existing `PublicPolicyContent` and `PublicSectionNav` components can be reused directly on `/framework` (with a minimal prop extension for section status badges). The `VersionStatusBadge` primitive is the pattern to clone for `SectionStatusBadge`.

The only novel technical work is: (1) deriving section status from existing schema at query time (no new columns) by joining `changeRequests` + `documentVersions`; (2) aggregating the what-changed log across multiple `documentVersions.changelog[]` JSONB arrays server-side; (3) adding the `isPublicDraft` column and corresponding tRPC mutation. All three are localized to new service/helper files and can be tested independently.

**Primary recommendation:** Follow the Phase 9 portal page as the structural template for `/framework`. Extract the what-changed aggregation into `framework-log.service.ts` immediately — both `/framework` and potential future consumers benefit, and it's the only logic requiring careful privacy-preserving unit tests.

### Pattern 5: `document.setPublicDraft` tRPC Mutation

**Mutation pattern (mirror `document.update`):**
```typescript
// Source pattern: src/server/routers/document.ts lines 189-222
setPublicDraft: requirePermission('document:update')
  .input(z.object({
    id: z.string().uuid(),
    isPublicDraft: z.boolean(),
  }))
  .mutation(async ({ ctx, input }) => {
    const [updated] = await db
      .update(policyDocuments)
      .set({ isPublicDraft: input.isPublicDraft, updatedAt: new Date() })
      .where(eq(policyDocuments.id, input.id))
      .returning()

    if (!updated) {
      throw new TRPCError({ code: 'NOT_FOUND', message: 'Document not found' })
    }

    writeAuditLog({
      actorId: ctx.user.id,
      actorRole: ctx.user.role,
      action: ACTIONS.DOCUMENT_SET_PUBLIC_DRAFT,
      entityType: 'document',
      entityId: input.id,
      payload: { isPublicDraft: input.isPublicDraft },
    }).catch(console.error)

    return updated
  }),
```

### Pattern 6: Section Status Derivation

**Service helper:** `src/server/services/framework-log.service.ts`

The derivation operates on data already in the database — no new columns required.

**Step 1 — Load all published `documentVersions` changelog arrays for the document:**
```typescript
// Source: src/db/schema/changeRequests.ts — documentVersions + changelog shape
const publishedVersions = await db
  .select()
  .from(documentVersions)
  .where(
    and(
      eq(documentVersions.documentId, documentId),
      eq(documentVersions.isPublished, true),
    ),
  )
```

**Step 2 — Build the Validated sectionId set:** Flatten all `changelog[]` entries across published versions and collect `affectedSectionIds`. A section is Validated if its sectionId appears in any entry.

```typescript
// Pure TypeScript — no Drizzle JSON query needed
const validatedSectionIds = new Set<string>()
for (const version of publishedVersions) {
  const entries = (version.changelog as ChangelogEntry[] | null) ?? []
  for (const entry of entries) {
    for (const sid of entry.affectedSectionIds) {
      validatedSectionIds.add(sid)
    }
  }
}
```

**Step 3 — Load open CRs for the document:**
```typescript
// Source: src/db/schema/changeRequests.ts — changeRequests table shape
const openCRs = await db
  .select({
    crId: changeRequests.id,
    sectionId: crSectionLinks.sectionId,
  })
  .from(changeRequests)
  .innerJoin(crSectionLinks, eq(changeRequests.id, crSectionLinks.crId))
  .where(
    and(
      eq(changeRequests.documentId, documentId),
      inArray(changeRequests.status, ['in_review', 'approved']),
    ),
  )

const underReviewSectionIds = new Set(openCRs.map((r) => r.sectionId))
```

**Step 4 — Build status map over sections:**
```typescript
export type SectionStatus = 'draft' | 'under_review' | 'validated'

export function getSectionPublicStatus(
  sectionId: string,
  validatedSectionIds: Set<string>,
  underReviewSectionIds: Set<string>,
): SectionStatus {
  if (validatedSectionIds.has(sectionId)) return 'validated'
  if (underReviewSectionIds.has(sectionId)) return 'under_review'
  return 'draft'
}
```

**Key insight:** The derivation is pure TypeScript after two DB queries. No JSONB `contains` operator is needed — the `changelog` field is always fetched as a TypeScript array from Drizzle's `$type<>()` binding. This is simpler and more type-safe than SQL-level JSONB array queries.

### Pattern 7: What-Changed Log Aggregation

**Also in `framework-log.service.ts`:**

```typescript
export interface PublicLogEntry {
  sectionTitle: string   // resolved title — never raw sectionId
  mergeDate: string      // ISO date string from publishedAt
  summary: string        // entry.summary ONLY — no crId, no feedbackIds
}

export function buildFrameworkLog(
  publishedVersions: Array<typeof documentVersions.$inferSelect>,
  limit = 20,
): PublicLogEntry[] {
  const entries: PublicLogEntry[] = []

  // Sort versions newest-first so log is in merge-date descending order
  const sorted = [...publishedVersions].sort(
    (a, b) =>
      new Date(b.publishedAt ?? b.createdAt).getTime() -
      new Date(a.publishedAt ?? a.createdAt).getTime(),
  )

  for (const version of sorted) {
    if (entries.length >= limit) break
    const changelog = (version.changelog as ChangelogEntry[] | null) ?? []
    const snapshot = (version.sectionsSnapshot as SectionSnapshot[] | null) ?? []
    const sectionTitleMap = new Map(snapshot.map((s) => [s.sectionId, s.title]))

    for (const entry of changelog) {
      if (entries.length >= limit) break
      const mergeDate = (version.publishedAt ?? version.createdAt).toISOString()

      // PRIVACY (PUB-05): only entry.summary — no crReadableId, no feedbackIds
      for (const sid of entry.affectedSectionIds) {
        const title = sectionTitleMap.get(sid)
        if (title) {
          entries.push({ sectionTitle: title, mergeDate, summary: entry.summary })
          if (entries.length >= limit) break
        }
      }
    }
  }

  return entries
}
```

**Privacy compliance (D-10):** Matches the existing pattern in `app/(public)/portal/[policyId]/changelog/page.tsx` lines 98-116. The `ChangelogEntry` type has `crId`, `crReadableId`, `crTitle`, and `feedbackIds` — none of these are included in `PublicLogEntry`.

**Data source note:** Both published versions (`isPublished = true`) and the section title map come from the same query already done for section status derivation. The page can pass the same `publishedVersions` result to both `getSectionPublicStatus` and `buildFrameworkLog` — no second DB round-trip.

### Anti-Patterns to Avoid

- **JSONB SQL operator for affectedSectionIds lookup:** Do NOT use Drizzle's `.arrayContains()` or raw `jsonb @>` operators to check sectionIds in changelog entries. The `changelog` column is fetched as a full TypeScript array via `$type<ChangelogEntry[] | null>()` — iterate in TypeScript, not SQL. SQL-level JSONB array queries are harder to test, less readable, and not needed when full version rows are already fetched.
- **Transaction usage:** Drizzle with Neon HTTP driver does NOT support `db.transaction()`. Use sequential awaits (established pattern throughout the codebase — see `version.service.ts` comment at line 250).
- **Importing Clerk or auth on public routes:** Public route components in `app/(public)/` must never import from `@clerk/nextjs` or call `auth()`. This is the Phase 9 invariant.
- **Using `policy:manage` permission:** This permission does not exist in the permission matrix. Use `document:update` for the `setPublicDraft` mutation.
- **Rendering `crReadableId`, `feedbackIds`, or `ownerId` on public pages:** PUB-05 binding. Only `entry.summary`, section titles, and merge dates may surface publicly.
- **Using drizzle-kit push for migration:** Never. All DDL is hand-written in `src/db/migrations/` and applied via `scripts/apply-migration-*.mjs`.

## Common Pitfalls

### Pitfall 1: `dynamic` export removed in Next.js 16 (conditional)

**What goes wrong:** Developer reads the Next.js 16 route segment config docs and sees `dynamic` was removed.
**Why it happens:** The route segment config index.md notes `dynamic` is removed "when Cache Components is enabled." This project does NOT enable `cacheComponents` in `next.config.ts`.
**How to avoid:** Confirm `next.config.ts` has no `cacheComponents: true`. Since it doesn't, `export const dynamic = 'force-dynamic'` continues to work exactly as in Phases 9, 19, 20. All existing public pages use it; the new pages follow the same pattern.
**Warning signs:** TypeScript error "Module has no exported member 'dynamic'" would indicate a config change — check `next.config.ts` first.

### Pitfall 2: `policy:manage` permission referenced in CONTEXT.md does not exist

**What goes wrong:** The `setPublicDraft` mutation uses `requirePermission('policy:manage')` — TypeScript will error because this key is not in the `PERMISSIONS` object.
**Why it happens:** D-02 in CONTEXT.md references `policy:manage` but this permission was never defined. The correct permission for document write operations is `document:update`.
**How to avoid:** Use `requirePermission('document:update')` for the `setPublicDraft` mutation. This is granted to `admin` and `policy_lead` — the correct audience.
**Warning signs:** TypeScript error on `requirePermission('policy:manage')` at compile time.

### Pitfall 3: Section status derivation misses the fallback case

**What goes wrong:** Sections not in the draft `documentVersions.sectionsSnapshot` (because they exist in `policy_sections` but no version was created yet) are not included in the status map.
**Why it happens:** D-07 specifies a fallback: if no draft version exists, read from live `policy_sections`. The status derivation must cover section IDs from whichever source is used.
**How to avoid:** Build the status map over the actual list of sections being rendered (from either `sectionsSnapshot` or `policy_sections`), not over a hardcoded set. The map defaults to `'draft'` for any unrecognized sectionId.

### Pitfall 4: JSONB changelog iteration produces duplicate log entries

**What goes wrong:** The `buildFrameworkLog` helper iterates per-entry per-affectedSectionId, producing duplicate entries when one changelog entry spans multiple sections.
**Why it happens:** If `entry.affectedSectionIds = ['sec-A', 'sec-B']`, the current design produces two log entries with identical summary but different section titles. This may be intentional (shows which sections were affected) but could look like a bug.
**How to avoid:** The design produces one entry per (section, version) pairing — this is documented behavior per D-09 ("grouped by section"). The `WhatChangedLog` component renders each entry independently; the limit of 20 applies to total entries, not unique CRs. This is correct behavior, but the unit test must verify it explicitly.

### Pitfall 5: `db.transaction()` used in migration runner

**What goes wrong:** Developer wraps multiple statements in a Drizzle transaction inside the migration runner.
**Why it happens:** Neon HTTP driver does NOT support transactions (`db.transaction()` is not available). The 0011 runner applies statements sequentially with `sql.query()`.
**How to avoid:** Migration 0012 has only one statement (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS`). The DO-block parsing logic from the 0011 runner script can be simplified or removed. Always use `sql.query(stmt)` directly.

### Pitfall 6: `public/research/` path conflicts with App Router

**What goes wrong:** A developer creates `app/(public)/research/page.tsx` AND places files at `public/research/` — assuming a conflict.
**Why it happens:** `public/` in Next.js serves files at `/` root. A file at `public/research/foo.pdf` is served at `/research/foo.pdf`. The App Router route `/research` serves from `app/(public)/research/page.tsx`. Both can coexist: the App Router handles `GET /research` (no trailing slash, no extension), while the file server handles `GET /research/consultation-research-report.pdf` (has extension).
**How to avoid:** Place the PDF at `public/research/consultation-research-report.pdf`. The App Router route is `app/(public)/research/page.tsx`. These do not conflict. (Verified in Next.js 16 docs: `public/` caching section confirms direct file serving without special config.)

### Pitfall 7: `PublicPolicyContent` prop extension breaks the portal page

**What goes wrong:** Adding `sectionStatuses` to `PublicPolicyContent` causes TypeScript errors on the portal page which passes no such prop.
**How to avoid:** Declare as optional (`sectionStatuses?: Map<...>`). Render badge inside a conditional (`{sectionStatuses && ...}`). The portal page import requires no change.

### Pitfall 8: Vitest drizzle mock too shallow for framework-log service

**What goes wrong:** Test of `buildFrameworkLog` fails because the db mock doesn't cover the `select().from().where()` chain used in `getSectionPublicStatus`.
**Why it happens:** Phase 18 established a deep mock stub pattern (see `evidence-request-export.test.ts`). The framework-log service tests should mock `@/src/db` with the same stub depth.
**How to avoid:** Follow the evidence-request-export test pattern: `vi.hoisted + vi.mock('@/src/db', ...)` with stubs for `select → from → innerJoin → where → returning`. Pure-function tests for `buildFrameworkLog` can skip DB entirely (pass pre-built arrays of version objects).

### Framework Page — Complete Data Loading Pattern

```typescript
// Source: mirrors app/(public)/portal/[policyId]/page.tsx + changelog/page.tsx
export const dynamic = 'force-dynamic'

import { db } from '@/src/db'
import { policyDocuments } from '@/src/db/schema/documents'
import { documentVersions, changeRequests, crSectionLinks } from '@/src/db/schema/changeRequests'
import { eq, and, desc, inArray } from 'drizzle-orm'
import { getSectionPublicStatus, buildFrameworkLog } from '@/src/server/services/framework-log.service'
import type { SectionSnapshot, ChangelogEntry } from '@/src/server/services/version.service'

export default async function FrameworkDetailPage({ params }) {
  const { policyId } = await params

  // UUID guard
  if (!UUID_REGEX.test(policyId)) notFound()

  // 1. Confirm document exists and is flagged for public draft
  const doc = await db.query.policyDocuments.findFirst({
    where: and(
      eq(policyDocuments.id, policyId),
      eq(policyDocuments.isPublicDraft, true),
    ),
  })
  if (!doc) notFound()

  // 2. Load most recent non-published draft version (D-07)
  const draftVersion = await db
    .select()
    .from(documentVersions)
    .where(
      and(
        eq(documentVersions.documentId, policyId),
        eq(documentVersions.isPublished, false),
      ),
    )
    .orderBy(desc(documentVersions.createdAt))
    .limit(1)
    .then((r) => r[0] ?? null)

  // 3. Load all published versions (for status derivation + log)
  const publishedVersions = await db
    .select()
    .from(documentVersions)
    .where(
      and(
        eq(documentVersions.documentId, policyId),
        eq(documentVersions.isPublished, true),
      ),
    )

  // 4. Build sections from draft snapshot OR live policy_sections (fallback per D-07)
  let sections: SectionSnapshot[]
  if (draftVersion?.sectionsSnapshot) {
    sections = (draftVersion.sectionsSnapshot as SectionSnapshot[])
      .sort((a, b) => a.orderIndex - b.orderIndex)
  } else {
    // Fallback: read live sections (import policySections schema)
    sections = await db
      .select({ ... })
      .from(policySections)
      .where(eq(policySections.documentId, policyId))
      .orderBy(asc(policySections.orderIndex))
  }

  // 5. Derive section statuses (two DB queries in service helper)
  const sectionStatuses = await getSectionPublicStatuses(policyId, publishedVersions)

  // 6. Build what-changed log (pure TS, no extra DB query)
  const logEntries = buildFrameworkLog(publishedVersions)

  // 7. Render
  return (
    <div>
      {/* ... layout mirrors /portal/[policyId] ... */}
      <PublicPolicyContent sections={sections} sectionStatuses={sectionStatuses} />
      <WhatChangedLog entries={logEntries} />
    </div>
  )
}
```

### `getSectionPublicStatuses` — Service Helper Signature

```typescript
// src/server/services/framework-log.service.ts
import { db } from '@/src/db'
import { changeRequests, crSectionLinks, documentVersions } from '@/src/db/schema/changeRequests'
import { eq, and, inArray } from 'drizzle-orm'
import type { ChangelogEntry, SectionSnapshot } from './version.service'

export type SectionStatus = 'draft' | 'under_review' | 'validated'

export async function getSectionPublicStatuses(
  documentId: string,
  publishedVersions: Array<typeof documentVersions.$inferSelect>,
): Promise<Map<string, SectionStatus>> {
  // Step 1: Build validated set from published changelogs (pure TS, no extra DB query)
  const validatedSectionIds = new Set<string>()
  for (const v of publishedVersions) {
    for (const entry of (v.changelog as ChangelogEntry[] | null) ?? []) {
      for (const sid of entry.affectedSectionIds) validatedSectionIds.add(sid)
    }
  }

  // Step 2: Load sections under active review CRs
  const openCRRows = await db
    .select({ sectionId: crSectionLinks.sectionId })
    .from(changeRequests)
    .innerJoin(crSectionLinks, eq(changeRequests.id, crSectionLinks.crId))
    .where(
      and(
        eq(changeRequests.documentId, documentId),
        inArray(changeRequests.status, ['in_review', 'approved']),
      ),
    )
  const underReviewSectionIds = new Set(openCRRows.map((r) => r.sectionId))

  // Step 3: Return function — caller maps section IDs to statuses
  return {
    get(sectionId: string): SectionStatus {
      if (validatedSectionIds.has(sectionId)) return 'validated'
      if (underReviewSectionIds.has(sectionId)) return 'under_review'
      return 'draft'
    },
  } as unknown as Map<string, SectionStatus>
  // NOTE: planner may prefer a plain function over a Map; adjust return type as needed
}
```

## Open Questions

1. **`getSectionPublicStatuses` return type: `Map<string, SectionStatus>` vs. plain function**
   - What we know: A `Map` is the most ergonomic for `PublicPolicyContent`'s `sectionStatuses?.get(section.sectionId)` pattern
   - What's unclear: The service helper may be cleaner returning a plain lookup function `(sectionId: string) => SectionStatus` that captures closure sets
   - Recommendation: Planner should choose; both work. The UI-SPEC uses `Map<string, 'draft' | 'under_review' | 'validated'>` — match that if possible.

2. **`buildFrameworkLog` entry shape when a section has no matching title in `sectionTitleMap`**
   - What we know: `sectionTitleMap` is built from `sectionsSnapshot` of the same version. If the changelog references a sectionId from a section that was later deleted, the title will be `undefined`.
   - What's unclear: Should the entry be skipped, or should it show a fallback like "Deleted section"?
   - Recommendation: Skip entries where `sectionTitleMap.get(sid)` is undefined (same behavior as `changelog/page.tsx` lines 101-103 which filters with `.filter(Boolean)`). This is already the PUB-05 safe default.

3. **`public/research/` placeholder PDF content**
   - What we know: D-14 says "a placeholder PDF ships with the phase so the link resolves"
   - What's unclear: Does "placeholder" mean a 1-page blank PDF, a text PDF saying "Coming soon", or the actual research content?
   - Recommendation: A minimal 1-page PDF titled "PolicyDash Research Report (placeholder)" with 3 lines of text is sufficient. The file just needs to be a valid PDF so browsers don't reject the download.
