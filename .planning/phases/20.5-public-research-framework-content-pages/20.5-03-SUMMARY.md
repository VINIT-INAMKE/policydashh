## Decisions Made

- **Backward-compatible prop extension via optional modifier + conditional render.** Extending PublicPolicyContent with `sectionStatuses?: Map<...>` and guarding the badge render with `{sectionStatuses && (...)}` is the lowest-risk way to reuse the component across portal (no badges) and framework (badges). The existing portal page file is zero-touch, confirmed by `git diff app/(public)/portal/[policyId]/page.tsx` (empty).
- **Switch primitive authored fresh, not vendored from shadcn/ui.** The project uses `@base-ui/react` (not `@radix-ui`) everywhere — Checkbox and Select both import from `@base-ui/react/checkbox` and `@base-ui/react/select`. The new Switch follows that pattern verbatim. Styling uses `data-checked` attributes (base-ui convention) and matches the project's existing ring/focus treatment.
- **PublicDraftToggle extracted to _components even though parent is already 'use client'.** Keeps the mutation + toast + invalidation concern isolated, makes the toggle trivially extractable to a future admin dashboard, and follows the phase's "minimal scope" guidance from the UI-SPEC.
- **Cloned the data-loading sequence in /framework/[policyId] rather than sharing a helper.** D-03 branching lives entirely in /framework; the detail route has its own `notFound()` + UUID guard path. A shared helper would have to thread `notFound` semantics through both sites and muddy the SSR flow.
- **Deferred pre-existing RBAC test failures rather than fixing inline.** The 2 `feedback-permissions.test.ts` failures pre-date this phase (commit `1648a46` widened the grant without updating the matrix). Fixing them would be a scope-unrelated RBAC test update.

Rather than stop, I authored a fresh Switch wrapper on top of the already-installed `@base-ui/react/switch` module, following the exact pattern used by the existing `components/ui/checkbox.tsx` (same project, same library, same styling conventions). The page is already `'use client'`, so strictly the toggle could have been inlined. I chose the extraction anyway for scope isolation (the toggle, mutation, and toast logic live in one testable module) — this is a clarity/maintenance improvement, not a behavioral change.
