# Phase 20.5: Public `/research` + `/framework` Content Pages - Context

**Gathered:** 2026-04-14 (auto-selected recommended defaults per user workflow preference)
**Status:** Ready for planning

<domain>
## Phase Boundary

Ship two public pages under `app/(public)/`:

1. **`/research`** — Static, unauthenticated content page rendering the research backing the policy: executive summary, current Indian landscape, key gap clusters, downloadable research report PDF, and a CTA to join consultation.
2. **`/framework`** — Public draft consultation surface for documents flagged `isPublicDraft: true`. Renders section content with per-section status badges (Draft / Under Review / Validated) derived from CR + version state, plus a "what changed" log aggregating recent CR merges per section (anonymized, no stakeholder identity, no CR IDs).

Both pages render without authentication via `proxy.ts` public-route whitelist.

**Out of scope:**
- Shared public shell (header/footer/nav) — Phase 21 delivers the minimal public shell + theme
- LLM consultation summary — Phase 21
- Full admin UI for toggling `isPublicDraft` beyond a minimal mutation path — can be a direct DB update or tiny workspace-side control; full admin UX is not this phase's goal
- Landing page `/` — deferred per ROADMAP.md
</domain>

<decisions>
## Implementation Decisions

### `isPublicDraft` flag (schema)
- **D-01:** Add `isPublicDraft BOOLEAN NOT NULL DEFAULT false` column to `policy_documents` via a new Drizzle migration. Applied with the Neon HTTP runner pattern (scripts/apply-migration-*.mjs) per canonical hand-written migration pattern from Phases 14/20.
- **D-02:** Expose a thin tRPC mutation `document.setPublicDraft` (admin-only via existing `policy:manage` permission) so the flag can be toggled without direct DB writes. A minimal toggle UI on the workspace policy detail page is in scope; a full admin dashboard is not.
- **D-03:** Querying `/framework` filters `policy_documents` by `isPublicDraft = true`. If zero match, render a friendly empty state ("No drafts currently under public consultation"). If one matches, render its detail inline at `/framework` (no redirect). If multiple match, render a simple card list at `/framework` and a detail view at `/framework/[policyId]`.

### Section status derivation (Draft / Under Review / Validated)
- **D-04:** Status is **derived at query time** from existing change-request and version state — no new `section_status` column. Rationale: keeps status always fresh, avoids schema churn, reuses Phase 5/6 data.
- **D-05:** Derivation rules (server-side, computed in a service helper `getSectionPublicStatus`):
  - **Validated** — the section appears in `affectedSectionIds` of at least one `changelog` entry in a *published* `documentVersions` row whose `isPublished = true`.
  - **Under Review** — the section has an open `changeRequest` with status `in_review` or `approved` (not yet `merged`) and no Validated mark above.
  - **Draft** — default when neither of the above holds (includes sections with only `open` or `rejected` CRs and sections with no CR activity).
- **D-06:** Status is computed once per page load from the latest draft version's `sectionsSnapshot` plus a single query over `changeRequests` + published `documentVersions` joined on `documentId`. No caching for v0.2 — recompute on every SSR render (`export const dynamic = 'force-dynamic'`).

### `/framework` content source
- **D-07:** Framework page reads the **most recent non-published `documentVersions` row** for the draft document as the "current draft snapshot" to render. If no draft version exists, fall back to rendering the live `policy_sections` rows directly (covers pre-version documents). Section content is rendered with the existing `PublicPolicyContent` component from `app/(public)/portal/[policyId]/_components/`, reused not duplicated.
- **D-08:** Section status badge is injected into the section header area next to the title. Layout mirrors `/portal/[policyId]`: sticky 240px left nav (desktop) + content column, plus a mobile section select.

### "What changed" log
- **D-09:** Renders the most recent **20 CR merges** for the document, grouped by section, sorted by merge date descending. Each entry shows: section title, ISO-formatted merge date (`MMM d, yyyy`), and the merge summary (or first changelog entry summary).
- **D-10:** Privacy rules inherit from Phase 9 PUB-05 precedent: **no** stakeholder identity, **no** CR readable IDs, **no** feedback IDs. Only `entry.summary` + affected section titles + merge date.
- **D-11:** Data source is `documentVersions.changelog[]` aggregated across all published versions for the document (same shape already used by `app/(public)/portal/[policyId]/changelog/page.tsx`). Helper extracted to `src/server/services/framework-log.service.ts` so both `/framework` and any future consumer can reuse it.
- **D-12:** The log renders as a dedicated section at the bottom of the framework page (not a separate subroute for v0.2). Title: "What changed". Empty state: "No changes merged yet."

### `/research` content
- **D-13:** Research content is **static React markup** for v0.2 — rendered inline in `app/(public)/research/page.tsx`. Sections match newDoc2.md §"Research page": executive summary, current Indian landscape, key gap clusters, download CTA, consultation CTA. No CMS, no DB table.
- **D-14:** Research report PDF lives at `public/research/consultation-research-report.pdf`. A `<a href="/research/consultation-research-report.pdf" download>` link renders the download CTA. A placeholder PDF ships with the phase so the link resolves; replacing the file post-ship is a plain asset drop.
- **D-15:** Page layout: single long-scroll column with semantic `<section>` headers; optional anchor-linked table-of-contents sidebar on desktop (≥ lg). No accordions, no client interactivity beyond anchor scroll.
- **D-16:** "Join consultation" CTA links to `/participate` (Phase 19 public intake form).

### Public route whitelisting
- **D-17:** Append `/research(.*)` and `/framework(.*)` to `createRouteMatcher` in `proxy.ts` using the Phase 19 append-at-end + comment-header pattern: a single `// Phase 20.5: PUB-06, PUB-07, PUB-08` comment block preceding the two new matchers, grep-auditable.
- **D-18:** No Turnstile or rate limiting on either GET page — both are read-only SSR content pages, not form endpoints.

### Design / theming
- **D-19:** Theming: use existing tokens/components (shadcn/base-nova, Tailwind v4). Pages inherit the current global CSS vars — the **policy-grade theme** (white base, dark slate typography, saffron/teal accent, document cards) is **Phase 21's responsibility**, not this phase. This phase ships functional structural markup using existing components; visual polish lands when Phase 21's theme is applied app-wide.
- **D-20:** Status badges reuse the existing CSS vars (`--status-cr-*`) and `VersionStatusBadge` pattern from `app/(workspace)/policies/[id]/versions/_components/`. Three variants: Draft (neutral), Under Review (amber), Validated (green).
- **D-21:** No shared public shell in this phase. Both pages render inside the existing `app/(public)/layout.tsx`, relying on whatever that layout currently provides. Phase 21 will upgrade the shell.

### UI spec prerequisite
- **D-22:** Per user workflow preference, `/gsd:ui-phase 20.5` MUST run before `/gsd:plan-phase 20.5`. The UI spec will concretize: research-page section visuals, framework-page section-header badge placement, what-changed log entry composition, and empty states.

<specifics>
## Specific Ideas

- Mirror the existing `/portal/[policyId]` layout on `/framework` — visitors should feel the two public surfaces are siblings.
- "What changed" log should read like a human-friendly diary of what the public can trust: date + plain-English summary + affected sections. No jargon, no internal IDs.
- Research page should feel print-friendly — a civil-society reader may want to print the executive summary and gap clusters.
- Section status badges should make the consultation state legible at a glance: a reader skimming the page should immediately see "which parts are still open for comment, which are under review, and which are final for this draft."
- Per newDoc2.md, both pages are static-feeling content surfaces — low interactivity, high legibility.
</specifics>
