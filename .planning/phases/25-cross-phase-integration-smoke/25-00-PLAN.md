**Fix 1 — TS18049 (20 errors across 2 files: 18 in workshop.ts + 2 in init.ts):**

The `enforceAuth` middleware at line 44 of `src/trpc/init.ts` checks `!ctx.userId || !ctx.user` and throws, but the `next({ ctx: { ...ctx, userId: ctx.userId, user: ctx.user } })` call does not narrow the types for downstream middleware. TypeScript still sees `user` as `typeof ctx.user` (i.e., the union including `null | undefined`).

Fix: Add a non-null assertion in the `enforceAuth` middleware's `next()` call so TypeScript understands the narrowing. Change lines 48-54 to:

```typescript
  return next({
    ctx: {
      ...ctx,
      userId: ctx.userId!,
      user: ctx.user!,
    },
  })
```

This is safe because the guard on line 45 throws before reaching `next()` if either is nullish.

**How the init.ts fix resolves all 18 workshop.ts errors:** All 18 errors in workshop.ts are `ctx.user` references inside `protectedProcedure` handlers. The `protectedProcedure` chains `enforceAuth` -> `touchActivity`, so all downstream handlers receive the context type produced by `enforceAuth`'s `next()` call. Once that `next()` narrows `user` to non-null (via `ctx.user!`), TypeScript propagates the narrowed type through the middleware chain to all handler callbacks. No changes to workshop.ts itself are needed.

**Fix for `src/server/routers/__tests__/evidence-request-export.test.ts`:**

In the `vi.mock('@/src/db', ...)` block (lines 52-81), add an `update` property to the `db` mock object:

```typescript
update: vi.fn().mockReturnValue({
  set: vi.fn().mockReturnValue({
    where: vi.fn().mockReturnValue({
      catch: vi.fn(),
    }),
  }),
}),
```

Add this after the existing `delete` mock (line 79), before the closing `}` of the `db` object.

**Fix for `tests/phase-20.5/set-public-draft-mutation.test.ts`:**

The current mock at line 20 is `dbMock: { update: vi.fn() }`. The `update` mock returns `undefined`, so `.set()` on the result fails. The test's `dbMock.update` is used both by the `touchActivity` middleware (which chains `.set().where().catch()`) and by the test's own `setPublicDraft` mutation (which chains `.set().where().returning()`).

Replace line 20 with a more complete mock:

```typescript
dbMock: {
  update: vi.fn().mockReturnValue({
    set: vi.fn().mockReturnValue({
      where: vi.fn().mockReturnValue({
        returning: vi.fn().mockResolvedValue([{ id: 'doc-1', isPublicDraft: true }]),
        catch: vi.fn(),
      }),
    }),
  }),
},
```

Since `dbMock.update` is called by both the mutation (needs `.returning()`) and the middleware (needs `.catch()`), both terminal methods must be on the same `where()` return object. The mock above satisfies both paths.

Also remove the duplicate `dbMock.update.mockReturnValue(...)` setup inside the test body (lines 39-47 of the test file) since the hoisted mock now covers it. If the test's `beforeEach` or test body re-configures `dbMock.update`, leave those in place but ensure they include `.catch` on the `where` return.
