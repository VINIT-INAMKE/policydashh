Function definition:
- `id: 'milestone-ready'`
- `name: 'Milestone ready — Cardano anchor'`
- `retries: 4` (per D-13 — Inngest built-in retry handles transient Blockfrost failures)
- `concurrency: { key: 'cardano-wallet', limit: 1 }` (per D-08 — one tx at a time)
- `triggers: [{ event: milestoneReadyEvent }]`

Handler with 5 steps + finalize:

**Step 1 `compute-hash`:** Load milestone from DB by `event.data.milestoneId`. Re-derive contentHash from DB data (do NOT trust event payload — research anti-pattern). Load linked entities (versions, workshops, feedback, evidence). Compute per-child hashes using same logic as markReady in milestone.ts (copy the hash computation pattern exactly — use same Date-to-ISO conversion). Build manifest. Call `hashMilestone({ manifest, metadata: { milestoneId, documentId, title, createdAt: ISO, requiredSlots } })`. Return `{ contentHash, milestoneId, documentId }`.

If milestone not found or status is `anchored`, throw `NonRetriableError` — don't retry.
If milestone status is `defining`, it means markReady hasn't persisted yet — also `NonRetriableError`.

**Step 2 `persist-hash`:** Update milestone row: `status: 'anchoring'`, persist `contentHash` from step 1 (the hash was already persisted by markReady as 'ready', but we re-derive and re-persist to ensure consistency). Write audit log with `ACTIONS.MILESTONE_ANCHOR_START`.

**Step 3 `check-existing-tx`:** Call `checkExistingAnchorTx(hashData.contentHash)` from `src/lib/cardano.ts`. Returns existing txHash string or null. This is VERIFY-08 Blockfrost pre-check layer.

**Step 4 `submit-tx`:** If `existing` from step 3 is non-null, return `existing` (skip submission — idempotent). Otherwise call `buildAndSubmitAnchorTx(hashData.contentHash, { project: 'policydash', type: 'milestone', hash: hashData.contentHash, milestoneId: hashData.milestoneId, timestamp: new Date().toISOString() })` per D-07. Return txHash.

**Step 5 `confirm-loop`:** Poll loop with MAX_ATTEMPTS = 20, sleep 30s between polls:
```typescript
let confirmed = false
let attempts = 0
const MAX_ATTEMPTS = 20
while (!confirmed && attempts < MAX_ATTEMPTS) {
  confirmed = await step.run(`confirm-poll-${attempts}`, async () => {
    return await isTxConfirmed(txHash)
  })
  if (!confirmed) {
    await step.sleep(`confirm-sleep-${attempts}`, '30s')
  }
  attempts++
}
```
CRITICAL: Each step ID MUST include `attempts` counter for unique Inngest memoization keys. The counter lives OUTSIDE step.run — this is correct per research Pitfall 2.

**Step `finalize`:** If confirmed: update milestone `txHash`, `anchoredAt: new Date()`, `status: 'anchored'`, `updatedAt: new Date()`. Write audit log `ACTIONS.MILESTONE_ANCHOR_COMPLETE`. Return `{ milestoneId, txHash, confirmed: true }`.

If NOT confirmed after MAX_ATTEMPTS: write audit log `ACTIONS.MILESTONE_ANCHOR_FAIL`. Send admin notification via `sendNotificationCreate({ userId: event.data.triggeredBy, type: 'cr_status_changed', title: 'Anchoring failed for milestone', body: 'Milestone anchoring timed out after 10 minutes. Use the Retry Anchor button on the milestone detail page.', entityType: 'milestone', entityId: event.data.milestoneId, createdBy: event.data.triggeredBy, action: ACTIONS.MILESTONE_ANCHOR_FAIL })` per D-14. Return `{ milestoneId, txHash, confirmed: false }`.

```typescript
// ---- retryAnchor (D-15) — re-emit milestone.ready for stuck anchoring state ----
retryAnchor: requirePermission('milestone:manage')
  .input(z.object({ milestoneId: z.string().uuid() }))
  .mutation(async ({ ctx, input }) => {
    const milestone = await loadMilestone(input.milestoneId)

    // Only allow retry when stuck in 'anchoring' state
    if ((milestone.status as MilestoneStatus) !== 'anchoring') {
      throw new TRPCError({
        code: 'CONFLICT',
        message: `Cannot retry anchor from state ${milestone.status} (must be anchoring)`,
      })
    }

    // Re-emit the milestone.ready event — milestoneReadyFn is idempotent
    // (D-08 concurrency lock + VERIFY-08 Blockfrost pre-check + DB UNIQUE)
    await sendMilestoneReady({
      milestoneId: milestone.id,
      triggeredBy: ctx.user.id,
      documentId:  milestone.documentId,
    })

    writeAuditLog({
      actorId:    ctx.user.id,
      actorRole:  ctx.user.role,
      action:     ACTIONS.MILESTONE_ANCHOR_START,
      entityType: 'milestone',
      entityId:   milestone.id,
      payload:    { retried: true },
    }).catch(console.error)

    return { status: 'retrying' as const }
  }),
```

Note: `retryAnchor` uses `await sendMilestoneReady` (not fire-and-forget) — the retry is user-initiated so we want to confirm the event was sent before responding. The anchor pipeline itself is still async.
